### 主要观察

L1 的主体表现是可用的：35 条里有 28 条通过，失败集中在两类问题上。

第一类是严格比对对字段别名过于敏感。33 条有 strict compare `fail`，但其中 28 条被语义复核判定为 `pass`。典型情况是 Gold Answer 使用 `value`、`type`、`n.elem_type` 这类字段名，而生成结果使用 `service_name`、`service_type`、`service_quality_of_service` 等更语义化的别名。实际返回值和行数正确，但 strict diff 将列名差异当作行差异。

第二类是真实字段遗漏，主要是生成 Cypher 漏掉了问题明确要求的 `ID` 字段。5 个 `issue_ticket_created` 都属于这种类型或相近类型：问题要求名称、ID、带宽、服务质量等字段，但生成结果只返回了部分字段。

另外有 2 条没有进入正常评测，状态为 `generation_failure_only`，原因是 cypher-generator-agent 返回了 `clarification_required`。这两条问题里的措辞包含“内部ID”“设备类型/网元类型”等可能引起 schema 语义澄清的表达。

### 初步判断

L1 的失败并不是 Cypher 语法或 TuGraph 连接问题。通过样本普遍能正确执行，并且大多数 strict fail 被语义复核修正为 pass。

当前 L1 的主要改进点有两个：

1. 评测侧应区分“字段别名不同但语义等价”和“字段缺失”。现在 strict compare 的 fail 数偏高，会制造大量需要语义复核的噪声。
2. 生成侧需要加强对“ID”字段要求的保真。只要问题里明确出现 `ID`，生成 Cypher 必须包含对应的 `id` 返回列，尤其是在同时要求名称、带宽、服务质量等字段时。
