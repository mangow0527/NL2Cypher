# Cypher 生成服务设计

## 1. 服务职责

Cypher 生成服务负责把自然语言问题转换成一条只读 Cypher，并把生成结果或生成失败报告提交给测试服务。它只负责“生成”，不负责执行 Cypher，不负责判断答案是否正确，也不负责创建知识修复建议。

正式生成入口：

- `POST /api/v1/qa/questions`

辅助诊断入口：

- `GET /api/v1/generator/status`
- `POST /api/v1/intents/recognize`
- `POST /api/v1/semantic/parse`

正式入口由 `CypherGeneratorAgentService` 编排。服务启动时会注入语义流水线、语义资产对齐检查、测试服务客户端、模型客户端和可靠投递队列。

## 2. 当前主链路

```text
自然语言问题
  -> 语义资产对齐检查
  -> 语义流水线解析
  -> 生成 Cypher 或生成失败报告
  -> 提交测试服务
```

这条链路的核心目标是：模型不能直接根据自然语言自由拼 Cypher。系统必须先把问题编译成受控的语义结构，再由确定性渲染器或受控模型生成 Cypher，最后通过生成前检查。

## 3. 语义资产对齐检查

语义资产对齐检查发生在正式生成之前。它检查生成系统依赖的资产是否可信。

检查内容：

- `semantic_layer.yaml` 只能引用 TuGraph 物理图谱中真实存在的点标签、边类型和属性。
- 语义层里的关系方向必须符合 TuGraph 的边约束。
- 知识目录里的 `schema.json` 必须和 TuGraph 物理图谱一致。
- `business_knowledge.md` 和 `few_shot.md` 中出现的正向字段引用、别名字段引用、关系模式，必须同时存在于 TuGraph 图谱和语义层。
- 反例、禁止项、错误示例中的旧字段不会被当成正向知识引用。

失败处理：

- 如果语义层、物理图谱和知识之间不一致，返回 `service_failed / semantic_contract_unaligned`。
- 如果知识目录缺失或必需文件缺失，返回 `service_failed / knowledge_context_unavailable`。

这一步的作用类似“开工前验货”。如果语义资产本身不可信，后面的生成结果也不能被信任，所以正式生成会直接停止。

## 4. 语义流水线

语义流水线负责把自然语言问题逐步编译成可校验的语义查询。

流程如下：

```text
自然语言问题
  -> 三阶段意图识别
  -> 底层槽位匹配
  -> 选择业务槽位规则
  -> 填充业务槽位并检查完整性
  -> 链接到语义层
  -> 语义层校验
  -> 生成受控语义查询
  -> 知识选择与抽取
  -> 渲染 Cypher
  -> 生成前检查
```

各步骤的作用：

- 三阶段意图识别：判断问题想要什么答案形态，例如明细、统计、分组、排名、存在性判断。识别顺序是规则、向量相似度、LLM 意图兜底。
- 底层槽位匹配：从问题中召回候选实体、关系、属性、指标、过滤条件、排序和数量限制。它只做候选抽取，不直接决定最终图谱落点。
- 业务槽位规则：根据意图判断这个问题必须具备哪些业务要素。
- 业务槽位完整性检查：如果必要信息缺失，停止生成，并返回可解释诊断。
- 语义层链接：把槽位候选绑定到正式图谱中的点、边、属性和指标。
- 语义层校验：确认绑定结果在语义层中被授权。
- 受控语义查询：把已校验的结果整理成统一的中间查询结构。
- 知识选择与抽取：根据受控语义查询从 RAG 中抽取相关知识片段，形成可追踪的最小知识上下文。
- Cypher 渲染：优先使用确定性渲染器生成只读 Cypher。
- 生成前检查：确认 Cypher 安全、只读，并且没有越过语义查询边界。

意图识别内部流程如下：

```text
自然语言问题
  -> 规则意图识别
  -> 向量相似度意图识别
  -> LLM 兜底意图识别
  -> intent accept
  -> 底层槽位匹配
```

LLM 兜底意图识别只输出意图 JSON，不生成 Cypher。它的作用是在规则和向量相似度都不能稳定接受时，补一次受分类体系约束的意图判断。如果 LLM 返回 `accept`，流水线会带着这个意图结果重新进入底层槽位匹配；如果返回 `clarify` 或输出不合法，流水线停止，不进入 Cypher 生成。

底层槽位匹配本身不依赖 intent 来决定抽取什么。它先从原始问题中召回所有可识别候选；随后业务槽位规则根据已接受的 intent 判断这些候选是否足够支撑当前查询任务。

## 5. 受控语义查询

`SemanticQuerySpec` 是 Cypher 生成前的受控中间结构。它不是 Cypher，而是系统内部的查询计划。

它包含：

- 查询类型：明细查询、指标查询、分组查询、排名查询、存在性查询。
- 参与实体、关系、返回字段、分组维度、指标、过滤条件。
- 排序、数量限制、输出别名。
- 来源意图、业务槽位规则编号、业务场景编号。

为什么需要它：

- 把“理解用户问题”和“生成 Cypher 语法”分开。
- 让确定性渲染器和模型兜底生成使用同一份约束。
- 让生成前检查知道什么是允许的，什么是越权的。
- 方便排查问题时看到系统到底理解成了什么。

明细类问题如果用户没有显式指定返回字段，系统会默认补 `id` 和 `name`。这样可以避免“业务槽位完整，但没有任何返回列”的情况。

## 6. 知识选择与抽取

知识选择与抽取发生在 `SemanticQuerySpec` 形成之后、Cypher 生成之前。它的职责不是重新理解自然语言，也不是替代语义层，而是根据已经校验过的查询结构抽取当前问题真正需要的知识上下文。

输入包括：

- 原始问题。
- 已接受的 intent。
- `SemanticQuerySpec`。

发送给 RAG 的语义线索包括：

- 查询类型。
- intent、业务槽位规则编号、业务场景编号。
- 参与实体、关系、属性和指标。

RAG 返回：

- `fragments`：被选中的知识片段列表。
- `prompt_context`：可放入模型提示词的知识上下文。
- `selection_trace`：知识选择过程的可追踪记录。
- `missing_knowledge_signals`：知识缺失或检索不可用信号。

这一步的原则是：

- 知识只能补充业务解释、查询习惯、few-shot、已验证查询和 Cypher 约束。
- 知识不能新增语义层没有授权的点标签、边类型或属性。
- 进入模型提示词的知识必须可以追踪来源和选择原因。
- RAG 不可用时，流水线记录知识选择不可用信号，不把不可信知识塞进生成过程。

当前实现中，确定性渲染器只依赖 `SemanticQuerySpec`，不依赖知识上下文。知识上下文主要提供给受控模型兜底生成使用；当确定性渲染器或生成前检查无法覆盖当前语义查询时，模型会同时看到用户问题、`SemanticQuerySpec`、渲染失败原因和已选择知识上下文。

## 7. Cypher 生成

Cypher 生成分两种路径。

第一种是确定性渲染：

- 输入是 `SemanticQuerySpec`。
- 输出是一条只读 Cypher。
- 适用于当前已支持的明细、统计、分组、排名和存在性查询。

第二种是受控模型兜底生成：

- 只有当确定性渲染器不支持当前语义查询时才会使用。
- 输入包含用户问题、`SemanticQuerySpec` 和渲染失败原因。
- 模型被明确禁止新增未授权的点标签、边类型和属性。
- 模型输出不会直接通过，仍然必须经过解析和生成前检查。

因此，模型不是默认自由生成器。它只能在语义边界内补足复杂生成能力。

这里的受控模型兜底生成不同于意图阶段的 LLM 兜底。意图阶段的 LLM 只判断 intent；Cypher 阶段的模型只有在已经形成 `SemanticQuerySpec` 且确定性渲染器无法覆盖时才会参与。

## 8. 生成前检查

生成前检查分两层。

第一层是基础安全检查：

- 不能是空输出。
- 不能有多条语句。
- 括号和字符串必须闭合。
- 不能包含写操作。
- 不能调用未允许的过程。
- 查询必须以允许的只读子句开始。

第二层是语义一致性检查：

- Cypher 中出现的点标签、边类型、属性必须来自 `SemanticQuerySpec`。
- 匿名节点标签也会检查。
- 节点属性字面量也会检查。
- Cypher 必须覆盖语义查询中的实体、关系、过滤条件、返回字段、分组维度、指标、排序、数量限制和输出别名。

常见生成失败原因：

- `unauthorized_schema_reference`：生成结果引用了未授权的图谱元素。
- `semantic_query_mismatch`：生成结果没有完整覆盖语义查询。
- `semantic_parse_rejected`：语义解析阶段未能形成可生成的查询。
- 其他解析或基础安全检查失败原因，例如 Markdown 包装、多语句、写操作等。

## 9. 成功提交的数据

生成成功后，服务会提交：

```json
{
  "id": "qa-001",
  "question": "查询 Gold 服务使用的隧道名称和时延",
  "generation_run_id": "cypher-run-001",
  "generated_cypher": "MATCH ... RETURN ...",
  "input_prompt_snapshot": "{ 语义流水线轨迹 }",
  "last_llm_raw_output": "MATCH ... RETURN ...",
  "generation_retry_count": 0,
  "generation_failure_reasons": []
}
```

说明：

- `generated_cypher` 是最终提交给测试服务的 Cypher。
- `input_prompt_snapshot` 在主链路中保存语义解析轨迹，不再是旧式全量知识提示词。
- `last_llm_raw_output` 在确定性生成路径中等于最终 Cypher；在模型兜底生成路径中保存模型原始输出。
- `attempt_no` 不由 Cypher 生成服务维护，而是由测试服务接收后分配。

## 10. 失败提交的数据

服务级失败示例：

```json
{
  "generation_status": "service_failed",
  "failure_reason": "semantic_contract_unaligned"
}
```

生成级失败示例：

```json
{
  "generation_status": "generation_failed",
  "failure_reason": "semantic_query_mismatch",
  "generation_failure_reasons": ["semantic_query_mismatch"],
  "gate_passed": false
}
```

区别：

- 服务级失败表示系统没有进入一次可信的生成过程，例如语义资产未对齐、知识目录不可用、模型调用失败。
- 生成级失败表示已经尝试生成，但结果没有通过解析、语义边界或安全检查。

## 11. 语义资产

当前使用的语义资产包括：

- `services/testing_agent/docs/reference/schema.json`：TuGraph 物理图谱的本地参考副本。
- `services/cypher_generator_agent/config/semantic_layer.yaml`：实体、关系、属性、指标和值映射的语义契约。
- `services/cypher_generator_agent/config/slot_dictionary.yaml`：底层槽位召回词典。
- `services/cypher_generator_agent/config/business_slot_schemas.yaml`：意图到业务槽位完整性的规则。
- `services/cypher_generator_agent/config/intent_*`：意图体系、规则、向量语料和评测集。
- RAG 服务中的知识片段：语义查询相关的 schema 解释、业务规则、few-shot、已验证查询和 Cypher 约束。

槽位词典里声明的实体属性，必须被语义层暴露。这样槽位匹配不会召回一个语义层不允许生成的字段。

## 12. 状态接口

`GET /api/v1/generator/status` 返回：

- 模型配置状态。
- 知识目录是否可用。
- 知识选择是否配置为 RAG。
- 语义资产对齐报告。
- 测试服务是否配置。

这个接口用于部署前和运行中诊断。正式生成入口仍会在每次处理问题前执行语义资产对齐检查。

## 13. 可靠投递

提交测试服务失败时，生成服务会把待提交内容保存到本地投递队列。

队列中可能保存两类内容：

- 成功生成的 Cypher 提交。
- 生成失败报告。

后台重试只处理当前契约的数据结构。4xx 类不可重试错误会进入死信区，避免无限重试。

## 14. 当前边界

- Cypher 生成服务不执行 Cypher。
- Cypher 生成服务不维护尝试次数。
- Cypher 生成服务不把失败自动归因成知识修复。
- 语义层是图谱 schema 权威。
- 知识检索、少样本示例和提示词只能补充上下文，不能创建新的图谱结构权威。
- 正式构建的服务使用语义流水线；只有测试代码显式关闭语义流水线时，才会走直接模型生成路径。
