# 提示词优化测试

## 说明

本文档保留三类测试结果：

1. `单阶段严格实验`
   - `CGS` 直接根据提示词生成 Cypher
   - 后续只升级了 `Testing Service` 的量化评测方式

2. `两阶段提示词实验`
   - `CGS` 先做语义拆解，再根据结构化语义表示生成 Cypher
   - 本轮重点测试的是：`两阶段提示词是否比单阶段更稳定地改善返回形状`

3. `意图理解 + 知识选择实验`
   - `CGS` 先做意图理解，识别题型、主实体、关系路径、过滤条件、聚合语义与返回形状
   - 再按题型选择真正需要的知识与对应 few-shot
   - 最后组装提示词生成 Cypher
   - 本轮重点测试的是：`显式做题型/意图理解，并按类型选择知识，是否比前两轮更进一步`

这三类实验的目的不同，因此结果需要并列保留，不能互相覆盖。

## 二、两阶段提示词实验

### 实验说明

这轮实验的核心变化不是瘦身提示词，而是：

1. 第一阶段先做 `semantic parsing`
2. 输出结构化语义表示
3. 第二阶段再根据：
   - 结构化语义表示
   - 原始生成提示词
   来生成 Cypher

这轮实验的目标是验证：

> `先语义拆解，再生成 Cypher` 是否能比单阶段更稳定地改善返回形状。

### 语义拆解包来源

本轮语义拆解包由实验手工维护，但内容参考了远端 `Knowledge Ops` 当前真实知识：

- `schema.json`

- `cypher_syntax.md`
- `few_shot.md`
- `system_prompt.md`

因此，这不是完全拍脑袋写的模板，而是：

> `以 KO 当前真实知识为依据，抽取出一层更适合 semantic parsing 的实验包。`

## 三、意图理解 + 知识选择实验

### 实验说明

这轮实验在前两轮基础上继续往前走一步：

1. 第一阶段不再只做“语义拆解 JSON”，而是显式做 `intent_understanding`
2. 结构化识别这几个固定语义：
   - `query_type`
   - `primary_entity`
   - `related_entities`
   - `relation_paths`
   - `filters`
   - `aggregation`
   - `ordering_limit`
   - `return_shape`
3. 第二阶段不再直接复用整包提示词，而是根据意图：
   - 选择真正相关的知识块
   - 选择对应问题类型的正例 / 反例 few-shot
4. 然后再组装生成提示词提供给 `CGS`

这轮实验的核心目标是验证：

> 让模型先知道“这是什么题型、应该怎么返回”，再按题型喂知识，是否比“统一大 prompt”更稳。

### 实验知识来源

本轮提示词不是由 `KO` 直接下发，而是实验侧根据远端 `KO` 当前真实知识目录重新组织：

- `schema.json`
- `business_knowledge.md`
- `cypher_syntax.md`
- `few_shot.md`
- `system_prompt.md`

此外，few-shot 的组织方式参考了文档：

- [图查询问题分类与FewShot设计.md](/Users/mangowmac/Desktop/code/NL2Cypher/services/query_generator_agent/docs/reference/图查询问题分类与FewShot设计.md)

### 意图理解 + 知识选择实验结论

这轮实验说明：

1. 显式做 `query_type + return_shape` 判断是有价值的
   - 尤其对 `link_dst_port` 这类关系关联查询，返回结构比上一轮更干净

2. 按题型挑知识与 few-shot 是可行的
   - 至少没有破坏已有稳定样本
   - 并且能让 prompt 更聚焦于当前问题

3. 对三条样本的收益并不均匀：
   - `link_dst_port`：生成结构进一步变好
   - `tunnel_bandwidth`：保持了两阶段实验的正向结果
   - `tunnel_type_count`：基本无变化

4. 当前这轮最大的未解问题仍然是：
   - `result_correctness.score = 0.0`
   - `result_set_f1 = 0.0`

也就是说，这轮实验进一步支持了一个判断：

> 现在 prompt / 意图理解改造已经能显著影响“应该怎么返回”，但还没有触及“为什么数据库执行结果与 golden 不一致”的更深层问题。

### 评测逻辑修复后的复测

复测时间：`2026-04-20`

本次复测先修正了 `Testing Service` 的两个评测问题：

1. `result_correctness` 对 Cypher 返回变量 alias 过于敏感
   - 例如 `RETURN a` 与 `RETURN t` 返回同一批节点对象时，旧逻辑会因为外层 key 不同判为不匹配
   - 修复后会对图节点对象做语义归一化，优先比较节点对象本身，而不是临时变量名

2. `schema_alignment` 的 label 提取误判
   - 旧逻辑会把 `[:LINK_DST]` 里的 `LINK_DST` 误当成节点 label
   - 修复后只从节点模式 `(n:Label)` 或 `(:Label)` 中提取 label

复测样本：

1. `semantic_intent_guided_eval_link_dst_port_20260420b`
2. `semantic_intent_guided_eval_tunnel_type_count_20260420b`
3. `semantic_intent_guided_eval_tunnel_bandwidth_20260420b`

复测结果：

| 样本 | 生成 Cypher | verdict | overall_score | result_correctness |
|---|---|---:|---:|---:|
| `link_dst_port` | `MATCH (l:Link)-[:LINK_DST]->(p:Port) RETURN l.id AS key, p LIMIT 5` | `pass` | `1.0` | `1.0` |
| `tunnel_type_count` | `MATCH (t:Tunnel) RETURN t.type AS group_key, count(t) AS total` | `pass` | `1.0` | `1.0` |
| `tunnel_bandwidth` | `MATCH (t:Tunnel) WHERE t.bandwidth >= 1 RETURN t LIMIT 10` | `pass` | `1.0` | `1.0` |

复测结论：

- 三条样本均在真实 TuGraph 上执行成功
- 三条样本的 `syntax_validity / schema_alignment / result_correctness / question_alignment` 均为 `pass`
- 之前 `result_correctness = 0.0` 的主要原因并不是生成 Cypher 查错数据，而是评测器对 alias 和返回 JSON 外形过于敏感
- 当前这组三样本下，`意图理解 + 知识选择` 生成的 Cypher 可以认为已经达到语义正确

## 四、当前总结

把三组实验放在一起看，当前更稳的结论是：

1. `单阶段严格实验`
   - 能较稳定地产生语法正确的 Cypher
   - 但在返回形状上经常过度投影

2. `两阶段提示词实验`
   - 对信息类问题的返回形状有正向作用
   - 这种改善在最新重跑中是可复现的，不是偶然波动

3. `意图理解 + 知识选择实验`
   - 对关系关联查询的返回结构进一步变干净
   - 说明显式 `query_type / return_shape` 有助于让 prompt 更聚焦
   - 在修复 Testing Service 的 alias / 返回 JSON 外形敏感问题后，三条样本均已在真实 TuGraph 上得到 `result_correctness = 1.0`

4. 当前最大未解问题仍然是：
   - 需要继续扩大样本集验证当前提示词策略是否泛化
   - 需要继续观察更复杂路径、多跳、排序和混合查询
   - 需要继续验证 Testing Service 的语义归一化是否覆盖更多等价返回形态

因此下一步如果继续优化，应该分开看三件事：

- 提示词是否能在更多题型上稳定改善返回形状与真实执行结果
- 按题型选知识和 few-shot 是否能继续稳定收益
- Testing Service 的语义归一化逻辑是否还能覆盖更多等价返回形态

## 五、后续建议

1. 在保持当前量化评测不变的前提下，继续测试两阶段 prompt 的：
   - `few-shot`
   - `return_shape`
   - 第二阶段约束表达

2. 重点观察这些指标是否继续提升：
   - `overall_score`
   - `schema_alignment.score`
   - `question_alignment.score`
   - `projection_match_score`

3. 后续应扩大评测集，重点覆盖：
   - 多跳路径
   - 排序 / Top-K
   - 多实体混合返回
   - 返回字段投影与对象返回混合场景
